# Review: Assistant research intelligence tools

## Verdict

The OpenSpec describes the correct assistant integration layer, but its original
ordering assumes that every upstream domain engine already exists. At the time of
implementation, market/sector context is available, while deep analysis,
watchlist synthesis, shortlist, scenario, and evidence engines are not all
available as stable public services.

The implementation therefore uses an adapter-first strategy:

```text
assistant intent
  -> deterministic plan
  -> local read-only tool contract
  -> persisted warehouse artifacts
  -> structured READY / PARTIAL / UNAVAILABLE payload
  -> grounded synthesis
  -> research answer audit
```

It does not fabricate unavailable engine output and does not move domain
calculation into the LLM.

## Critique of the original design

### 1. Dependency order was underspecified

The spec depends on outputs from several earlier OpenSpecs, but it did not define
how the assistant should behave while one of those engines is unavailable.

Resolution:

- stable tool contracts are implemented now;
- adapters use existing persisted warehouse artifacts;
- missing upstream artifacts produce explicit `PARTIAL` or `UNAVAILABLE` results;
- later engines can replace adapter internals without changing assistant plans.

### 2. Integration and domain computation were too easy to conflate

`assistant-research-intelligence-tools` should own taxonomy, orchestration,
groundedness, policy, and audit. It should not become the permanent home for all
technical-analysis, shortlist, scenario, or historical-evidence algorithms.

Resolution:

- tool implementations are split into symbol, watchlist, scenario, and evidence
  modules;
- the public tool names remain stable;
- each payload records methodology, artifact references, freshness, caveats, and
  missing data;
- richer upstream engines can be adopted through the same tool facade.

### 3. Groundedness needs two gates, not one

The design placed a groundedness validator before answer synthesis. That catches
missing tool payloads but cannot detect claims invented by the synthesizer.

Resolution:

```text
pre-synthesis gate
  checks required tool outputs and required payload fields

post-synthesis gate
  checks evidence-backed claim vocabulary, missing-data disclosure,
  caveat coverage, and research-only language
```

### 4. Caveats are not the same as missing data

A valid persisted snapshot can have methodological caveats without being missing.
Treating every warning as missing data causes false failures.

Resolution:

- warnings influence groundedness status (`WARN`);
- only null payloads, explicit missing-data lists, or partial/unavailable statuses
  require `missing_data` disclosure in the answer.

### 5. Auditability must include both database and file observability

Tool traces alone show what ran, but they do not capture whether the final answer
passed groundedness and policy checks.

Resolution:

- `research_answer_audit` stores intent, tools, artifact refs, freshness,
  groundedness, policy status, and caveats;
- `RESEARCH_ANSWER_AUDITED` is emitted to file observability best-effort;
- prompt and raw answer content are not copied into the audit metadata.

### 6. Shortlist and scenario workflows require a stricter policy boundary

These workflows are the most likely to drift from research description toward
action guidance.

Resolution:

- deterministic research-language validation is mandatory;
- the tool outputs use conditional monitoring language;
- shortlist output is framed as research prioritization;
- scenario branches require future confirmation and human review;
- the assistant policy still rejects broker, account, allocation, and execution
  tool paths.

## Implementation boundary

Implemented in this change:

```text
new intents and classifier examples
deterministic plans
safe local tool registry entries
warehouse-grounded adapter tools
per-intent synthesis templates
pre/post groundedness validation
research-only language validation
research answer audit persistence
focused and end-to-end tests
documentation
```

Not claimed as completed by this change:

```text
full multi-timeframe deep-analysis engine
full market-regime builder execution
full shortlist persistence engine
full scenario-plan persistence engine
full setup event-study engine
```

Those remain independent domain capabilities. The assistant adapters expose
structured missing evidence until their richer implementations are available.

## Risk controls

- No arbitrary SQL is generated by the model.
- No network or filesystem tool is added to assistant plans.
- All assistant tools are declared in the central immutable tool policy.
- Data provisioning remains a deterministic precondition for symbol analysis.
- Returned metrics must exist in deterministic tool payload keys.
- Research answers require caveats and disclose partial or unavailable evidence.
- Shortlist and scenario wording passes the research-language validator.
