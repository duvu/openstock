# Review: Auto data provisioning gap analysis

## Verdict

The current codebase has the required manual pipeline primitives, but it does not yet meet the requirement that symbol analysis automatically loads missing data.

Current state:

```text
Manual data pipeline:              partially complete
Symbol analysis from persisted data: complete enough
Automatic one-symbol provisioning: not implemented
Assistant precondition provisioning: not implemented
```

## Existing capabilities

### Manual ingestion and build commands

The CLI already supports:

```text
vnalpha sync symbols
vnalpha sync ohlcv --symbols FPT --start ... --end ...
vnalpha sync index --symbol VNINDEX --start ... --end ...
vnalpha build canonical --symbol FPT
vnalpha build features --symbols FPT --date ...
vnalpha score --symbols FPT --date ...
```

### vnstock-service client

The HTTP client already supports:

```text
get_symbols()
get_equity_ohlcv()
get_equity_quote()
get_index_ohlcv()
get_provider_health()
get_provider_capabilities()
```

### Persisted analysis artifacts

The analysis flow already uses:

```text
market_ohlcv_raw
canonical_ohlcv
feature_snapshot
candidate_score
daily_watchlist
ingestion_run
```

## Blocking gaps

### 1. `/explain SYMBOL` does not provision data

`/explain` currently reads `candidate_score` through the `candidate.explain` tool.

If no score exists, it returns a warning asking the user to run `vnalpha score --date ...` manually. It does not attempt to sync OHLCV, build features, or score the symbol.

### 2. `/scan` does not provision data

`/scan` reads `daily_watchlist`. If empty, it asks the user to run scoring manually. This may be acceptable for full-universe scanning, but it does not satisfy one-symbol explain/compare workflows.

### 3. Assistant tools are read-oriented

The assistant tool allowlist contains read tools such as:

```text
candidate.explain
candidate.compare
watchlist.scan
quality.get_status
lineage.get_symbol_lineage
```

There is no deterministic tool or pre-execution hook that ensures data artifacts exist before read tools run.

### 4. Feature builder detects missing inputs but does not fetch them

`build_features()` can skip symbols because canonical data is missing or insufficient. It can also warn when the benchmark is missing. But it does not call the ingestion layer to fetch missing data.

### 5. No freshness policy exists

The system currently lacks a central policy for:

```text
how much lookback history is required
when data is considered stale
whether to refresh already-existing rows
how to handle weekends/holidays
what to do when the target date has no trading bar
```

### 6. No one-symbol lock exists

If two requests ask for the same symbol/date simultaneously, they could both attempt to sync/build/score the same artifacts. A local lock or idempotency guard is needed.

### 7. Observability does not explain provisioning decisions

Existing domain events cover feature build and watchlist generation, but there is no clear event trail for:

```text
why data was considered missing
what was fetched
what was reused
what failed
whether the final state is ready or partial
```

## Required behavior

For a single-symbol analysis request, the system should prefer the cheapest valid path:

```text
candidate_score fresh enough
  -> use it immediately

candidate_score missing but feature_snapshot exists
  -> score symbol only

feature_snapshot missing but canonical data sufficient
  -> build features for symbol, then score symbol

canonical data missing/stale
  -> sync symbol OHLCV, build canonical, build features, score symbol

benchmark missing/stale
  -> sync benchmark OHLCV, build canonical, build features, score symbol
```

## Design risks

### Risk: auto-refresh becomes slow

One-symbol `/explain` should not trigger full-universe work. It should only sync/build the requested symbol and benchmark.

### Risk: ambiguous dates

If the target date is not a trading day, the system should use the latest bar on or before target date and report `as_of_bar_date` clearly.

### Risk: provider returns empty data

Empty provider data must produce `PARTIAL` or `FAILED` with clear warnings, not a silent empty analysis.

### Risk: hiding stale data

If stale data is used, the answer must include freshness information.

### Risk: overloading assistant tool permissions

The LLM should not decide to mutate local artifacts. Ensure-data should be a deterministic precondition in the execution layer, not an arbitrary plan step generated by the LLM.

## Recommended architecture

```text
DataAvailability service
  -> check symbol master
  -> check candidate_score
  -> check feature_snapshot
  -> check canonical_ohlcv history window
  -> check benchmark canonical_ohlcv
  -> run minimal sync/build/score actions
  -> return EnsureDataResult
  -> log DATA_ENSURE_* events

User-facing integration
  -> /explain SYMBOL
  -> /compare SYMBOL1 SYMBOL2
  -> assistant candidate.explain/candidate.compare pre-execution hook
```

## Readiness after implementation

Expected result after implementation:

```text
Manual sync still supported: yes
One-symbol auto analysis: yes
Assistant natural-language auto provisioning: yes
Full-universe automatic refresh: optional, not default
Data freshness visible in answer: yes
Closed-loop debugging visibility: yes
```
