# Assistant research-intelligence tools

## Purpose

The assistant research-intelligence layer turns persisted warehouse artifacts into bounded, auditable research answers. It does not grant the model direct access to SQL, files, network browsing, unrestricted code execution, broker systems, accounts, allocation, or order execution.

## Supported workflows

| Intent | Deterministic primary tool | Purpose |
|---|---|---|
| `deep_analyze_symbol` | `analysis.deep_symbol` | Multi-block review of one symbol using scores, features, levels, context, freshness, and lineage |
| `review_market_regime` | `market.get_regime` | Read persisted market-regime context |
| `review_sector_strength` | `sector.get_strength` | Read persisted ranked sector context |
| `summarize_watchlist_deep` | `watchlist.summarize_deep` | Summarize classes, setups, sectors, quality, and risk flags |
| `generate_shortlist` | `shortlist.generate` | Produce a deterministic research-priority list |
| `generate_research_scenario` | `scenario.generate_research_plan` | Produce conditional confirmation, neutral, and invalidation states |
| `review_setup_evidence` | `evidence.get_setup_history` | Read persisted setup outcome evidence |

Related context tools include `sector.get_symbol_alignment`, quality tools, and lineage tools.

## Runtime flow

```text
user prompt
  -> deterministic request policy
  -> intent classifier
  -> deterministic PlanBuilder
  -> central assistant tool policy
  -> AssistantExecutor
  -> bounded deterministic tool payloads
  -> pre-synthesis grounding validation
  -> task-aware model routing and synthesis
  -> post-synthesis grounding validation
  -> research-language policy validation
  -> deterministic fallback when validation fails
  -> research_answer_audit persistence
  -> final AssistantAnswer
```

## Deterministic planning

The LLM classifies intent but does not choose arbitrary tools. `PlanBuilder` maps each supported intent to a fixed set of policy-approved tool calls.

Each plan declares:

```text
intent
steps and arguments
required permissions
required artifacts
assumptions
```

Unknown, unsafe, or incomplete requests produce a refusal or clarification path rather than an improvised tool call.

## Tool trust boundary

Research-intelligence tools are registered in the shared local registry and declared in the central tool policy.

Allowed properties:

```text
bounded inputs
structured ToolOutput payloads
persisted warehouse reads
deterministic calculations
trace persistence
explicit missing-data and caveat fields
```

Disallowed properties:

```text
raw SQL generated by the model
filesystem access
external browsing
unrestricted Python or shell execution
broker, account, allocation, margin, or execution functions
```

`data.fetch` remains unavailable to autonomous assistant plans. One-symbol data readiness is handled as a deterministic executor precondition for analysis tools.

## Tool payload contract

Research tool payloads should expose the applicable subset of:

```text
available or status
symbol / setup / as_of_date
structured research metrics
freshness
lineage
quality
methodology
artifact_refs
missing_data
caveats
policy metadata
```

A tool must not silently invent unavailable fields. Missing upstream artifacts must be represented explicitly.

## Synthesis templates

Every research intent has a template defining:

```text
required payload fields
allowed research wording
required caveats
missing-data disclosure rule
```

The template and a bounded list of valid source references are included in the synthesis context.

The expected model response envelope is:

```json
{
  "summary": "...",
  "basis": "...",
  "risks_caveats": "...",
  "tool_trace_summary": "...",
  "missing_data": [],
  "grounded_source_refs": [],
  "research_metadata": {}
}
```

The model may use only source references listed in `valid_grounded_source_refs`.

## Groundedness gates

### Before synthesis

The runtime verifies that:

```text
all planned tools produced outputs
the primary payload is structured
required fields are present when data is available
partial payloads disclose missing data
```

A missing tool output or an undisclosed contract violation prevents the LLM call.

### After synthesis

The runtime verifies that:

```text
source references belong to the executed plan or tool artifact payloads
numeric claims are represented in the structured payloads
missing artifacts are disclosed
basis is non-empty
risks and caveats are non-empty
```

Dates are excluded from numeric-claim matching. Percentage and collection-size claims are matched against normalized payload values.

## Research-language policy

All research answers are checked for execution-oriented wording. Shortlist and scenario workflows also require an explicit research-only or not-a-recommendation disclaimer.

Allowed framing includes:

```text
persisted research context
research priority
monitoring condition
confirmation
neutral state
invalidation
requires human review
```

The system does not return personalized action instructions or certainty claims.

## Deterministic fallback

For research-intelligence intents, the system can build a final answer directly from structured tool payloads when:

```text
the model provider is unavailable
model output is invalid JSON
source references are unsupported
numeric claims are not grounded
required caveats or policy wording are missing
```

The fallback answer is subject to the same groundedness and policy validators. If the fallback fails, the request fails closed with `SynthesisError`.

Existing market and sector caveat-first checks continue to fail closed for unsafe or misleading wording.

## Research answer audit

Successful research-intelligence answers create one row in `research_answer_audit`.

Stored fields:

```text
research_answer_audit_id
assistant_session_id
created_at
intent
tools_json
artifact_refs_json
dataset_freshness_json
groundedness_status
groundedness_json
policy_status
policy_json
caveats_json
correlation_id
```

The assistant answer stores `research_answer_audit_id` in `research_metadata`.

The audit table intentionally stores compact metadata rather than the full prompt or full tool payload. The standard assistant session, LLM trace, and tool trace tables retain their existing responsibilities.

## Observability

Best-effort file audit logging emits:

```text
RESEARCH_ANSWER_AUDITED
```

Metadata includes intent, tool names, groundedness status, policy status, and artifact-reference count. Raw prompt and answer content are not copied into this event.

## Model routing

Research synthesis passes task metadata to the model router:

```text
deep symbol analysis       -> reasoning
multi-symbol comparison    -> reasoning
shortlist generation       -> reasoning
research scenario          -> reasoning
large watchlist synthesis  -> reasoning
ordinary answers           -> default
```

Routing does not change the tool trust boundary.

## Failure behavior

| Failure | Behavior |
|---|---|
| Planned tool output missing | Fail before synthesis |
| Tool payload partial with explicit missing data | Continue with disclosure requirements |
| LLM unavailable or invalid JSON | Use deterministic fallback |
| Unsupported source or numeric claim | Use deterministic fallback |
| Unsafe market/sector wording | Fail closed |
| Fallback fails validation | Raise `SynthesisError` |
| Research audit cannot be persisted | Raise `SynthesisError`; do not mark session successful |
